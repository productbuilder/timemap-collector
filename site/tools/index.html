Tools

Open Collections Protocol defines how collections can be published and discovered on the web. The tools around it make that model practical.

Some tools help people create and publish collections. Some help people view them. Others help discover, register, or connect collections across domains and institutions.

Together, they form an open ecosystem: different tools can work with the same published collections without depending on one central platform.

Collection Manager

Create, edit, organize, and publish collections

Collection Manager is the main tool for preparing and publishing collections with Open Collections Protocol.

It helps collection owners:

  • create or organize a collection
  • edit item metadata
  • manage media and assets
  • connect storage providers
  • generate and publish collection outputs

For non-technical users, Collection Manager is the practical entry point. It turns the protocol into something you can actually work with.

For developers and integrators, Collection Manager is also an embeddable component. That means it can be used not only as a standalone tool, but also inside websites, portals, collection systems, and other applications.

Over time, it should support a flexible attribute and configuration API so it can adapt to different publishing contexts and interface needs.

Collection Browser

View and explore published collections

Collection Browser is the viewing and exploration counterpart to Collection Manager.

It helps people:

  • open a published collection
  • browse items and media
  • explore metadata
  • use collection views inside websites, portals, and public interfaces

Like Collection Manager, Collection Browser can work both as a standalone tool and as an embeddable component.

That makes it useful for:

  • institutional websites
  • public collection portals
  • thematic exhibitions
  • regional or national aggregation sites
  • custom collection interfaces

Once a collection is published through the protocol, many different interfaces should be able to present it.

Indexer

Discover and organize collections

An Indexer finds published collections, reads their manifests, and helps make them searchable and discoverable.

Indexers can:

  • crawl collection endpoints
  • follow domain discovery files
  • organize collections for search or aggregation
  • connect related collections across publishers

This is similar to how search engines work on the web: they do not own the websites they index, but they help people find and navigate them.

Registrator

Register or announce collections

A Registrator helps publishers register, announce, validate, or curate collection endpoints.

Depending on the ecosystem, a Registrator might:

  • register a collection or domain discovery endpoint
  • validate that a collection is published correctly
  • support curated registries
  • make collections easier to find and trust

This is especially useful when collections need to be gathered across institutions, communities, or thematic networks.

TimeMap

Explore collections through time and context

TimeMap is a presentation layer built on top of published collections.

It can help people explore material through:

  • time
  • place
  • theme
  • cross-collection relationships

Where Collection Browser is a general-purpose browsing interface, TimeMap can become a more interpretive or exploratory interface built on the same published resources.

Tools and components

Standalone tools and embeddable building blocks

Collection Manager and Collection Browser are not only standalone tools. They can also be used as embeddable web components.

That means they can be integrated into:

  • heritage portals
  • museum or archive websites
  • collection management interfaces
  • custom apps
  • public browsing environments

This matters because the protocol is not tied to one interface. The same published collection should be usable in many contexts.

A flexible attribute and configuration API will make these components easier to adapt to different needs, layouts, permissions, and user journeys.

Why this matters

The protocol provides the shared publishing model. The tools make that model usable.

That combination matters:

  • collection owners get simpler ways to publish and manage collections
  • viewers get better ways to explore them
  • developers get reusable components and interfaces
  • institutions stay free to integrate the tools into their own environments

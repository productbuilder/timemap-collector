Storage Providers

A collection needs somewhere to live online.

Open Collections Protocol does not require one central platform or one mandatory hosting model. Collections can be published through different kinds of storage providers, as long as the result is a stable web surface with clear URLs, manifests, and assets.

For most practical uses, two storage models matter most:

  • Git-based providers
  • S3-compatible object storage providers

Why storage matters

The protocol is about how collections are published and discovered, not about forcing one internal system.

That means you can choose storage based on what fits your situation:

  • how technical your team is
  • how much media you have
  • how much versioning you need
  • how much hosting flexibility you want

The important thing is that the published result remains stable and understandable:

  • a collection manifest at a URL
  • item detail URLs where needed
  • media assets at URLs
  • optional domain discovery through DCD

Git providers

Versioned publishing with familiar web workflows

Git-based providers are often a good fit for collections that are published as files and updated through a versioned workflow.

Examples include:

  • GitHub
  • GitLab
  • Forgejo
  • Gitea

Git-based publishing is especially useful when you want:

  • transparent version history
  • simple file-based publishing
  • static site hosting workflows
  • easier review of changes
  • a straightforward path for small to medium collections

This can be a practical option for:

  • documentation-like collections
  • curated public collections
  • pilot projects
  • institutions that want to publish in a transparent and portable way

Git-based publishing can also work well with static hosting and CDN delivery.

S3-compatible storage

Scalable object storage for media and assets

S3-compatible storage is a strong fit when collections include larger numbers of media assets or need more scalable object storage.

Examples include:

  • AWS S3
  • Cloudflare R2
  • Backblaze B2 with S3 compatibility
  • MinIO
  • Wasabi

S3-style storage is especially useful when you want:

  • scalable media hosting
  • reliable asset delivery
  • separation between manifests and media
  • CDN-friendly storage patterns
  • more flexibility for larger collections

This is often a good choice for:

  • image-heavy collections
  • audiovisual material
  • larger institutional collections
  • systems where media storage and metadata publication are handled separately

Git or S3?

Choose the model that fits your publishing needs

A simple rule of thumb:

Choose Git-based storage when:

  • you want a versioned, file-first workflow
  • your collection is relatively manageable in size
  • you want transparent publishing and easy review of changes

Choose S3-compatible storage when:

  • you have a large number of media files
  • you need scalable asset hosting
  • you expect heavier storage and delivery needs

In many cases, the best answer is a combination:

  • store collection manifests and structured publication outputs in Git
  • store larger media assets in S3-compatible storage
  • publish both as stable URLs on the web

Storage and the protocol

The protocol does not care whether your collection comes from Git, S3, a CMS, or another internal system.

What matters is the public result:

  • stable URLs
  • a machine-readable collection manifest
  • item detail resources where needed
  • media assets that can be fetched independently
  • optional domain-level discovery through DCD

This keeps the protocol flexible while still keeping the public web surface consistent.

Why this matters for non-technical publishers

Many people want to publish a collection without running a full custom platform.

That is why storage choice matters so much.

A good storage setup should help you:

  • get online more easily
  • keep control of your published files
  • avoid unnecessary infrastructure complexity
  • stay portable over time

The protocol supports that by letting collections be hosted in practical, familiar ways rather than forcing everyone into one system.

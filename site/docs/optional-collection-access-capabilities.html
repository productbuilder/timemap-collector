Optional Collection Access Capabilities

This layer adds scalable access patterns for very large collections. It is optional and should not be required for baseline publication.

1. When to use these capabilities

  • Use the core protocol only when a single manifest is practical.
  • Add access capabilities when full inline item listings become too large to serve or maintain efficiently.

2. Capability types

  • Pagination: split item listings into pages.
  • Filtering: support constrained item access by type, date range, or other stable fields.
  • Incremental sync: expose updatedAt plus query parameters like updatedAfter.
  • Query templates: declare URL templates so clients can build valid requests consistently.

3. Recommended optional manifest shape

{
  "id": "rotterdam-harbor",
  "title": "Rotterdam Harbor Collection",
  "protocolVersion": "1.0",
  "capabilities": {
    "pagination": true,
    "filtering": true,
    "incrementalSync": true
  },
  "itemsPageUrl": "https://museum-example.org/collections/rotterdam/items?page=1",
  "queries": {
    "itemSearchUrlTemplate": "https://museum-example.org/collections/rotterdam/items{?page,pageSize,type,from,to,updatedAfter}"
  }
}

This shape is additive. Collections without these fields remain fully valid.

4. Example paginated response

{
  "items": [
    {
      "id": "map-123",
      "title": "Harbor map, 1898",
      "type": "image",
      "thumbnailUrl": "thumbs/123.jpg",
      "detailUrl": "items/123.json",
      "updatedAt": "2025-09-01T12:00:00Z"
    }
  ],
  "nextPageUrl": "https://museum-example.org/collections/rotterdam/items?page=2"
}

5. Interoperability guidance

  • Keep canonical item URLs stable, regardless of pagination or filtering views.
  • Treat page/query endpoints as access surfaces, not primary identities.
  • Document supported parameters and sort order for consistent synchronization.
  • If mirror endpoints exist, publish canonical manifest and item URLs clearly.

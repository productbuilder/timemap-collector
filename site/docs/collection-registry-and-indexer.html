Collection Registry and Collection Indexer

These tools are related but distinct. Registry tracks what is known; Indexer processes what is known into discovery/search outputs.

Collection Registry (public term)

Collection Registry is the public-facing tool name. Registrator is the internal architectural term used for this module.

  • register publishers and collection endpoints
  • track validation and indexing state
  • federate or link with other registries
  • preserve provenance for imported records

Collection Indexer

  • fetch manifests and item records from known collection URLs
  • create searchable/browsable derived indexes
  • refresh on schedule or by event/job triggers
  • feed Browser, TimeMap, and aggregator experiences

How they work together

A single deployment may host both Registry and Indexer in early phases. The architecture still keeps them conceptually separate:

  • Registry = authoritative list and status records.
  • Indexer = derived processing and retrieval outputs.

Practical deployment baseline

  • SQLite-backed persistence for Registry data
  • scheduled jobs for indexing refresh
  • JSON export/import and optional published registry.json
  • growth path from local to institutional to federated deployments

Open Collections Protocol

Open Collections Protocol is a lightweight way to publish collections on the open web. It treats a collection as a first-class web resource, not as a row inside one central database. The architectural model is LinkedCollections, and the domain-level discovery mechanism is Domain Collections Discovery (DCD).

1. Why collections need a web-native protocol

Many collections are still trapped inside custom platforms, local databases, or one-off APIs. That makes long-term access, preservation, reuse, and cross-institution discovery harder than it needs to be.

Databases are useful, and many organizations should keep using them internally. But requiring a complex backend before anything can be published raises the barrier, especially for smaller museums, archives, libraries, and community groups.

Open Collections Protocol lowers that barrier. It defines a simple published web surface based on HTTP and JSON. You can still use databases, CMSs, or indexing systems behind the scenes; the protocol simply avoids making them mandatory for publication.

2. The core idea: collections as published web resources

The easiest way to understand the protocol is to compare it to a normal website:

Web concept Open Collections Protocol concept
Website Collection
index.html collection.json (manifest)
Web page Item record
Image/file Media asset URL
Search engine Indexer
Browser Collection Browser/Viewer

A collection is something you publish at a URL. The manifest is the entry point. Items and media are addressable resources that other tools can fetch, cache, index, and reuse.

3. How it works

The protocol uses a summary-first model. A manifest gives lightweight item summaries. Detailed metadata is loaded only when needed, and media files are loaded separately.

This keeps publication simple, keeps manifests small and cacheable, and helps browsers render quickly without downloading everything up front. 

{
  "id": "rotterdam-harbor",
  "title": "Rotterdam Harbor Collection",
  "items": [
    {
      "id": "map-123",
      "title": "Harbor map, 1898",
      "type": "image",
      "thumbnailUrl": "thumbs/123.jpg",
      "detailUrl": "items/123.json"
    }
  ]
}
{
  "id": "map-123",
  "title": "Harbor map, 1898",
  "date": "1898",
  "assets": [
    {
      "role": "full",
      "url": "media/123.jpg",
      "mimeType": "image/jpeg"
    }
  ]
}

4. Discovery and domain authority (DCD)

Domain Collections Discovery (DCD) is a small discovery file at /.well-known/collections.json. It lets a domain declare which collections it considers authoritative.

The collection files can live on the same server or elsewhere. The important point is that the domain-level declaration gives a trust anchor. 

{
  "domain": "museum-example.org",
  "collections": [
    {
      "id": "rotterdam-harbor",
      "title": "Rotterdam Harbor Collection",
      "manifestUrl": "https://cdn.museum-example.org/collections/rotterdam/collection.json"
    }
  ]
}

5. Collections of collections

Collections are first-class resources, so one collection can reference another collection. This enables federation and reuse without copying data.

In practice, this supports thematic curation across institutions, national or regional aggregations, and recursive composition where a collection can include subcollections. 

{
  "id": "north-sea-maps",
  "title": "North Sea Map Collections",
  "items": [
    {
      "id": "rotterdam-maps",
      "type": "collection",
      "url": "https://museum-a.org/collections/rotterdam/collection.json"
    },
    {
      "id": "antwerp-maps",
      "type": "collection",
      "url": "https://archive-b.be/collections/antwerp/collection.json"
    }
  ]
}

6. The ecosystem around the protocol

Open Collections Protocol is not one centralized platform. It is a protocol and ecosystem pattern. Different tools can operate on the same published resources.

  • Publishing layer: manifests, item files, and media published over HTTP.
  • Identity layer: canonical URLs for public identity, optional content hashes for verification.
  • Discovery layer: DCD files, registries, and indexers that discover and catalog collections.
  • Presentation layer: browsers, viewers, timelines, and other end-user applications.

Typical ecosystem actors include Collector (publishes), Registrator (registers/curates discovered endpoints), Indexer (crawls and builds search), and Browser/Viewer tools such as TimeMap. 

Domain
  ├─ /.well-known/collections.json (DCD)
  └─ collection.json
       ├─ items/*.json
       └─ media/*

Indexer / Browser / Aggregator
  └─ consume published URLs and build different experiences

7. Why this matters for heritage organizations

For museums, archives, libraries, local history groups, and community projects, this model supports incremental adoption. You can start with a simple, static, openly published collection and improve it over time.

  • Lower technical barrier for publication.
  • Domain-owned authority rather than forced platform lock-in.
  • Portability and long-term preservation of published artifacts.
  • Easier discovery, aggregation, and reuse across institutions.
  • No requirement to move everything into one shared backend.

8. Why this matters for developers

The protocol is HTTP-native, JSON-first, and static-host friendly. It is easy to test locally, cache, crawl, and integrate with existing tooling.

  • No required central backend for publication.
  • Works with plain static hosting and CDN distribution.
  • Straightforward to build editors, crawlers, registries, browsers, and importers.
  • Supports progressive enhancement: start simple, layer richer services later.

If your stack uses databases or CMS systems, keep using them. The protocol does not dictate internal storage. It defines the published web contract that others can consume reliably.

9. Identity in simple terms

Inside a collection, local IDs such as map-123 are useful. Publicly, the main identity is the canonical URL of the manifest, item, or media file.

Optional content hashes can be added when needed for integrity checks, deduplication, and version-aware workflows. In short: URL tells you where a resource is; hash can tell you exactly what bytes it is.

10. Relationship to existing standards

Open Collections Protocol is a lightweight baseline. It does not replace IIIF, JSON-LD, schema.org, RDF, or open data catalogs.

  • IIIF: can be linked for rich image delivery and viewers.
  • JSON-LD / schema.org: can be embedded or referenced for semantic interoperability.
  • Linked Data / RDF: can be added in advanced workflows without blocking basic publication.
  • Open data portals: can ingest or point to published collection manifests.

The priority is practical adoption and open publication first. Richer semantic and domain standards can be layered on top.

11. What the protocol includes — and what it does not

Included in the core: manifest publication, item and media URLs, lightweight discovery (DCD), and interoperable consumption by independent tools.

Not required in the core: central governance service, mandatory triplestores, mandatory IIIF stacks, mandatory commercial licensing systems, or any single vendor platform.

Optional extensions can exist for richer provenance, analytics, licensing, access control, and commercial services. The protocol boundary stays minimal and open.

12. Closing summary

Open Collections Protocol makes collection publishing work like the web itself: simple entry points, addressable resources, independent hosting, and many possible consumers. LinkedCollections provides the architectural model. DCD provides domain-level discovery and authority.

For heritage institutions and developers alike, this creates a practical path to publish once, stay in control, and participate in a broader open ecosystem without waiting for a single centralized platform.

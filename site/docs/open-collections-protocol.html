Open Collections protocol

The protocol defines how collections are published as stable web resources. It keeps the core simple for static hosting while allowing optional capabilities for larger and more dynamic collections.

Protocol scope

A valid collection is centered on a collection.json manifest plus item detail files and media files. The manifest acts as an index that points to richer data instead of trying to carry a full record store in a single file.

  • Core protocol: stable URLs, manifest, item detail references, and web-friendly publication.
  • Optional capabilities: pagination, filtering, and incremental sync fields for large collections.
  • Additive design: optional fields never invalidate a simple core collection.

Minimal core manifest

The core structure intentionally stays small and readable.

{
  "id": "harbor-collection",
  "title": "Harbor Collection",
  "description": "Historic harbor maps",
  "protocolVersion": "1.0",
  "canonicalUrl": "https://museum-example.org/collections/harbor/collection.json",
  "license": "CC BY 4.0",
  "rightsStatement": "https://rightsstatements.org/vocab/InC/1.0/",
  "items": [
    {
      "id": "harbor-map-001",
      "title": "Harbor Map 001",
      "type": "image",
      "thumbnailUrl": "thumbs/harbor-map-001.jpg",
      "detailUrl": "items/harbor-map-001.json",
      "updatedAt": "2025-09-01T12:00:00Z"
    }
  ]
}

Top-level fields

  • id: stable collection identifier.
  • title, description: human-facing collection description.
  • protocolVersion: compatibility marker (for example "1.0").
  • canonicalUrl: canonical public URL for the manifest.
  • license, rightsStatement: rights metadata.
  • items: lightweight item summaries linking to detail files.
  • capabilities and queries: optional scalability and query declarations.

Item summaries and item detail files

Keep item summaries lightweight in the manifest (id, title, type, thumbnailUrl, detailUrl, optional updatedAt). Put richer metadata into dedicated item detail files.

This split keeps top-level indexes fast to fetch while still supporting deep metadata for downstream indexing and specialized applications.

Identity model: local IDs, URLs, and hashes

The protocol supports layered identity. Use local IDs for authoring convenience, canonical URLs for global identity, and optional content hashes for verification and deduplication.

  • URL answers: where the resource is.
  • Hash answers: what the resource contains.
  • Together they support mirrors, integrity checks, and long-term portability.

Optional capabilities for large collections

Large collections may need paged listing and filtered retrieval. The protocol supports this additively through capability declarations and query templates. 

{
  "capabilities": {
    "pagination": true,
    "filtering": true,
    "incrementalSync": true
  },
  "itemsPageUrl": "https://museum-example.org/collections/harbor/items?page=1",
  "queries": {
    "itemSearchUrlTemplate": "https://museum-example.org/collections/harbor/items{?page,pageSize,type,updatedAfter}"
  }
}

Collections without these fields remain fully valid. Optional capability support is for scale, not for baseline compatibility.

Discovery and ecosystem layering

Publishing and discovery are intentionally separate. Owners publish collections at their own URLs; registries and indexers can then aggregate those collections for search and multi-collection browsing.

  • Publishing layer: manifest, item files, and assets via normal HTTP hosting.
  • Identity layer: canonical URLs and optional content hashes.
  • Discovery layer: registry and indexer services for aggregation.
  • Presentation layer: manager, browser, and custom consumers.

Extension and schema direction

Extensions should be additive and namespaced to prevent collisions. The protocol should continue to mature with maintained JSON schemas for manifests, item records, and optional capability blocks.

Guiding principle: keep the base manifest easy to publish and index, then layer richer features without breaking the core model.

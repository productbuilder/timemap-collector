Open Collections Protocol

Publish collections like the web

Open Collections Protocol is a lightweight, web-native way to publish, discover, and reuse collections as open resources with stable URLs, simple manifests, and domain-based authority.

Instead of locking collections inside one platform or requiring a complex backend just to get online, the protocol treats a collection as a published web resource: it has its own URL, manifest, and discoverable presence.

Collections should be easier to publish

Today, many collections are trapped in custom platforms, database-heavy systems, or one-off APIs. That makes portability, reuse, preservation, and discovery harder than they should be.

Smaller institutions, local history projects, and community publishers are often hit hardest by setup and maintenance overhead.

Databases are often useful behind the scenes. But publication should not depend on database complexity as a prerequisite. Open Collections Protocol defines the published web surface while allowing any internal storage model behind it.

A collection works like a website

Think of a collection the way you think of a website. A collection manifest plays the role of an index.html page: it introduces the collection and links to what is inside. Item pages are item detail resources, media files are web assets, indexers act like search engines, and browsers/viewers let people navigate what has been published.

This mental model keeps publishing understandable for both heritage teams and developers while remaining compatible with richer backend systems.

A simple publishing model

1. Publish a collection manifest

Give the collection a stable web URL and a machine-readable manifest.

2. Link item and media URLs

Each item and asset has a stable address that tools and people can reference.

3. Add domain discovery (optional)

Publish a Domain Collections Discovery (DCD) document to advertise collections from a domain.

4. Let the ecosystem consume

Indexers, browsers, aggregators, and other tools can use the same published surface.

Not another central platform

Open Collections Protocol is a shared protocol and ecosystem pattern, not a single system where everyone must store their data. Publishers keep control, and different tools can work with the same collection.

The result is more portability, easier reuse, and a stronger path for long-term preservation.

Built for federation and reuse

Collections can reference other collections. That supports thematic curation, regional aggregation, and cross-institution publishing without requiring every record to be copied into one central platform.

Designed for real collection owners

Museums, archives, libraries, local history groups, and community curators can publish with a lower technical barrier while keeping control of their collections and references.

  • Lower barrier to publish and maintain
  • Stable references for citation and preservation
  • Easier discovery across institutions and domains
  • Less dependence on any single vendor platform

Simple on the surface, flexible underneath

For developers, the model is straightforward: HTTP + JSON, stable URLs, and crawlable resources that work well with static hosting and caching.

You can generate the published surface from a CMS, a database, a static pipeline, or hand-authored files. Progressive enhancement remains possible, because the protocol defines publication outputs, not storage internals.

Many tools, one shared web surface

LinkedCollections describes the architecture behind the protocol, while Domain Collections Discovery (DCD) supports discovery across domains. Around that core, ecosystem tools can take on different roles.

Publisher
Publishes manifests and item/media URLs.
Collector
Creates and manages collection publication outputs.
Registrator
Helps register or announce published collections.
Indexer
Crawls and indexes collection resources for discovery.
Browser / Viewer / TimeMap
Presents collections for public exploration.

These roles are ecosystem options, not mandatory central services.

A lightweight baseline, not a replacement

This approach can coexist with IIIF, JSON-LD/schema.org, RDF/Linked Data, and open data catalogs. Open Collections Protocol provides a lightweight baseline for publication and discovery, while richer standards can still be layered where needed.

Open, portable, and web-native

A collection should be able to exist online as an ordinary web resource first. More complex systems can still sit behind the scenes when needed.
